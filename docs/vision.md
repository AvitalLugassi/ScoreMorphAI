# סקירת הפרויקט - ScoreMorph-AI

## סקירת הפרויקט (Project Overview)

אני בונה מערכת עיבוד מוזיקלי מבוססת בינה מלאכותית (AI) כפרויקט אקדמי.

**חשוב:** מטרת הפרויקט אינה לשחזר את השיר המקורי בדיוק ואינה להפריד כל כלי נגינה מההקלטה.

**המטרה האמיתית:** לחלץ את המידע המוזיקלי המרכזי מתוך קובץ אודיו שהועלה, ולייצר עיבוד חדש בהתאם להעדפות המשתמש.

---

## זרימת משתמש עיקרית (Main User Flow)

### שלב 1: העלאה וניתוח
1. המשתמש מעלה קובץ אודיו (MP3/WAV)
2. המערכת מנתחת את השיר
3. המערכת מחלצת:
   - **מלודיה מרכזית** (Main melody)
   - **מהלך אקורדים** (Chord progression)
   - **קצב** (Tempo / BPM)
   - **סולם מוזיקלי** (Musical key - אם אפשר)

### שלב 2: בחירת העדפות
המשתמש בוחר העדפות עיבוד:
- **סגנון מוזיקלי** (קלאסי, ג'אז, פופ, רוק וכו')
- **רמת קושי** (מתחילים / בינוניים / מתקדמים)
- **מספר כלי נגינה**
- **סוגי כלי נגינה**
- **מספר קולות** (Voices)

### שלב 3: יצירה וייצוא
1. המערכת מייצרת עיבוד חדש המבוסס על:
   - המידע המוזיקלי שחולץ
   - העדפות המשתמש
2. המערכת מייצאת:
   - **MIDI** (לתוכנות סימון)
   - **MusicXML** (סטנדרט בינלאומי)
   - **תווים** (PDF)

## החלטת עיצוב חשובה (Important Design Decision)

### אנחנו **לא** עושים:
- ❌ לא משחזרים את ההקלטה המקורית
- ❌ לא מזהים כל כלי נגינה בנפרד
- ❌ לא יוצרים תמלול (טראנסקריפציה) מושלם של השיר המלא

### אנחנו **כן** עושים:
- ✅ מחלצים את **המבנה המוזיקלי הבסיסי** של השיר:
  - מלודיה
  - אקורדים
  - קצב (Tempo)
- ✅ משתמשים במידע זה כ**בסיס ליצירת עיבוד חדש**

## ארכיטקטורה מתוכננת (Planned Architecture)

### 1. melody_extractor.py
**אחראי על:** חילוץ המלודיה המרכזית מתוך האודיו

**כלים אפשריים:**
- Demucs (בידוד שירה אופציונלי)
- Basic Pitch
- librosa

**פלט (Output):** מלודית MIDI

---

### 2. chord_detector.py
**אחראי על:** זיהוי מהלך האקורדים

**כלים אפשריים:**
- librosa
- ספריות לזיהוי אקורדים (chord recognition)

**דוגמה לפלט:** 
```
C → Am → F → G
```

---

### 3. tempo_detector.py
**אחראי על:**
- זיהוי BPM
- מידע על הקצב (Tempo)
- סולם מוזיקלי (אופציונלי - Key detection)

**כלים אפשריים:**
- librosa

**דוגמה לפלט:**
```
BPM = 90, Key = C Major
```

---

### 4. arrangement_engine.py
**רכיב הליבה של הפרויקט**

**קלט (Input):**
- מלודיה
- אקורדים
- BPM
- העדפות משתמש

**פלט (Output):** עיבוד חדש

**התאמה ל:**
- בחירת כלי הנגינה
- רמת הקושי
- הסגנון המוזיקלי

**דוגמה:**
```
קלט:  מלודיה, אקורדים, סגנון: פסנתר, רמה: מתחילים
פלט:  עיבוד פסנתר פשוט
       - יד ימין: מנגנת מלודיה
       - יד שמאל: מנגנת אקורדים פשוטים
```

---

### 5. score_generator.py
**אחראי על יצירת:**
- MusicXML
- MIDI
- תווים ב-PDF

**כלים אפשריים:**
- music21
- MuseScore (בעבודה)

---

## שלב הפיתוח הנוכחי (Current Development Stage)

### MVP (מוצר מינימלי בר-קיימא)
אנחנו בונים כרגע את ה-MVP.

### סדר עדיפויות:
1. ✅ **חילוץ מלודיה** (melody_extractor.py)
2. ✅ **זיהוי אקורדים** (chord_detector.py)
3. ✅ **זיהוי קצב** (tempo_detector.py)
4. ⏳ **מנוע עיבוד** (arrangement_engine.py)
5. ⏳ **יצירת תווים וקבצים** (score_generator.py)

### מה **לא** בעדיפות כרגע:
- ❌ צד-לקוח (Frontend)
- ❌ בסיס נתונים (Database)
- ❌ אימות משתמשים (Authentication)
- ❌ פריסה (Deployment)

**התמקדות:** צינור עיבוד המוזיקה המרכזי (Core music processing pipeline)

## הנחיות כתיבת קוד (Coding Guidelines)

### כללים כלליים:
1. ✅ השתמש **ב-Python**
2. ✅ כתוב **קוד נקי ומודולרי**
3. ✅ כל קובץ אחראי **על משימה אחת בלבד**
4. ✅ הוסף **הערות ותיעוד פונקציות** (Docstrings)
5. ✅ העדף **פתרונות פשוטים** לפני הכנסת מודלים מורכבים של AI

### ארכיטקטורה:
- קוד **מודולרי** המאפשר **הרחבות עתידיות**
- **הוכחת היתכנות (Proof of Concept)** קודם, אפליקציה מלאה אח"כ

### דפוס הדיוק (Pattern of Accuracy):
- התחל עם פתרון פשוט
- בדוק אם זה עובד
- הרחב והשפר בהדרגה

## דוגמה לערך (Example Value)

### קלט:
```
קובץ אודיו: "song.mp3" (Beatles - Let It Be)
```

### פלט (מחולץ):
```
melody = [C4, D4, E4, F4, G4, ...]
chords = ["C", "Am", "F", "G", ...]
tempo = 87 BPM
key = "C Major"
```

### בקשת משתמש:
```
סגנון: קלאסי
רמה: בינוניים
כלים: עליל, כלי קשת (Violin)
```

### פלט (עיבוד חדש):
```
פרטיטורה ל-Violin:
- Part 1: Melody + Harmonies
- Part 2: Arpeggios/Counter-melody

מקביל ל: Mozart-style arrangement
```

```
📄 let_it_be_violin_arrangement.pdf
```

## תכנית הפרויקט

**הפרויקט הזה הוא הוכחת היתכנות (POC) אולם מובנה בצורה המאפשרת הפכה לאפליקציה מלאה בעתיד.**

כל קובץ service כתוב כדי להיות:
- ✅ Testable (בר-בדיקה)
- ✅ Expandable (בר-הרחבה)
- ✅ Replaceable (בר-החלפה)

בעתיד ניתן להחליף את רכיבי ה-"placeholder" ב-implementations אמיתיים של ML/AI מבלי להשפיע על שאר הקוד.
